trait ACSet[A]

To work with ACSets in a typed way, we use the following encoding into Scala's type system.

To work with ACSets in a typed way, we use the following encoding into Scala's type system.

A typed ACSet is essentially a newtype wrapper around a BareACSet, i.e.

case class WeightedGraph[T](acset: BareACSet)

Then to use this newtype wrapper, one declares a trait implementation that has an Iso between the newtype wrapper and a BareACSet (essentially witnessing that the type is in fact just a wrapper around a BareACSet) and then also provide a schema that should be used for all operations.

For instance,

given weightedGraphACSet[T]: ACSet[WeightedGraph[T]] with
 val bare = GenIso[WeightedGraph[T], BareACSet]
 val schema = Schema(
   E(),
   V(),
   Src(),
   Tgt(),
   Weight[T]()
 )

This trait then provides extension methods for the newtype wrapper that pass in the schema to the functions on the BareACSet.

case class BareACSet(nextId: Int, obs: Map[Ob, Set[Entity]], homs: Map[AbstractHom, Map[Entity, Entity]], attrs: Map[AbstractAttr, Map[Entity, Any]])

This is the data of an acset, represented dynamically.

This is the data of an acset, represented dynamically.

A BareACSet by itself does not have a schema; a schema must be passed in for each of its methods.

Type checks against this schema are done at runtime to validate the methods.

For simplicity, we use unindexed hashmaps to store the action of the ACSet on the morphisms in the schema, and we use unique integers for each element in the category of elements of the ACSet. This massively simplifies the implementation, and we are not concerned about performance because these methods are performed at a time scale of user input, not at a time scale of algorithms.

If this assumption changes, we may have to revisit this decision, but this is good for now.

Also, notice that this is purely functional; methods that modify the acset instead return a new acset.
